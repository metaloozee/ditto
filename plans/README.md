## Verified target workflow

1. User sends a message (e.g. "Center the div").
2. Worker authenticates; ensures project sandbox is awake/hydrated (`ensureProjectSandbox` ā†’ restore backup if needed).
3. Creates or reuses a D1 `workspace_sessions` row (conversation).
4. Persists the user message in D1.
5. Creates a sandbox **shell** session (`createSession`) for this agent run.
6. Writes a job file into the sandbox (avoids shell injection); runs the PI harness via `execStream`.
7. Harness opens/resumes a PI agent session under `/workspace/.ditto/sessions/<conversationId>.jsonl` with coding tools.
8. Harness prints NDJSON events on stdout; worker parses `execStream` and forwards as client SSE.
9. On completion: persist assistant message in D1, `createBackup` workspace, update `projects.sandboxBackup`.
10. Client renders streaming text/tool events.

## Findings considered and rejected

- **Mount R2 over `/workspace` for persistence**: rejected ā€” Cloudflare docs recommend backups for project dirs; repo already chose backups; mounts can overlay image-seeded files.
- **Run the agent in the Worker process** (not container): rejected ā€” tools need a real FS/shell; Worker CPU limits; product is sandbox-centric.
- **Use tRPC subscriptions for streaming**: rejected ā€” Fetch SSE over a dedicated route matches Sandbox `execStream` and TanStack Start handlers already used for `/api/*`.
- **App-layer file locks for concurrent agents**: deferred explicitly per product request; documented in plan 003 only.
