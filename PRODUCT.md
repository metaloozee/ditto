# Product

## Register

product

## Users

Ditto serves people who want to create, import, understand, and evolve React-oriented web applications with AI assistance. Primary users range from low-code builders who need guided setup and plain-language explanations to experienced developers who want a faster web-based harness for scaffolding, inspecting, chatting with, and changing project code.

Users are typically in a build workflow: starting a new project, importing a GitHub repository, reviewing the generated app structure, chatting with an AI agent about changes, and applying those changes inside a sandboxed project environment.

## Product Purpose

Ditto is an AI harness on the web for creating and importing projects, primarily React applications. It gives each project an AI-assisted workspace where users can chat, scaffold, make changes, review context, and move from idea to working application with less local setup friction.

Success means users can confidently start from scratch or import an existing repo, understand what the AI is doing, provide project context, and make iterative changes without feeling trapped in an opaque code-generation black box.

## Brand Personality

Modern, clean, minimal. Ditto should feel calm, capable, and precise: a focused workspace for making software rather than a flashy AI toy. The tone should be helpful and direct, with enough warmth for non-experts and enough rigor for developers to trust it.

## Anti-references

Avoid noisy “AI playground” dashboards, generic SaaS hero/metric patterns, cluttered IDE chrome, gimmicky neon gradients, heavy glassmorphism, overdecorated cards, and interfaces that make AI activity feel magical but uninspectable.

Ditto should not look like a traditional desktop IDE crammed into a browser, nor like a landing-page template wearing product controls. It should prioritize task clarity, project confidence, and legible state over spectacle.

## Design Principles

1. **Make the project feel tangible.** Users should always understand which project, repo, environment, model, and branch they are working with.
2. **Guide without patronizing.** Non-experts need clear choices and consequences; developers need fast paths and accurate technical labels.
3. **Keep AI actions inspectable.** Planning, scaffolding, edits, environment setup, and errors should be visible enough to build trust.
4. **Prefer calm density.** The UI should support complex workflows without visual noise: consistent controls, restrained color, readable hierarchy, and purposeful empty states.
5. **Design for iteration.** Importing, chatting, editing, previewing, and revising should feel like one continuous loop rather than disconnected modals.

## Accessibility & Inclusion

Target WCAG AA. Prioritize keyboard-first navigation, visible focus states, readable contrast, accessible names for icon controls, clear disabled/loading/error states, and reduced-motion support. The product should remain understandable for users with less coding experience while preserving accurate terminology for technical users.
